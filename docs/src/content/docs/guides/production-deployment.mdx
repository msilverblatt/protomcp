## dev vs run

| | `dev` | `run` |
|--|-------|-------|
| Hot reload | Enabled (file watch) | Disabled |
| Intended for | Local development | Production |

```sh
# Development
protomcp dev tools.py

# Production
protomcp run tools.py
```

In `run` mode, protomcp starts the tool process and does not watch for file changes. Reload requires restarting the protomcp process.

## Choosing a transport

For production, choose the transport that matches your deployment topology:

| Transport | Flag | When to use |
|-----------|------|-------------|
| stdio | `--transport stdio` | MCP client spawns protomcp as a subprocess (most common) |
| HTTP SSE | `--transport sse` | Remote or web-based MCP clients, single connection |
| Streamable HTTP | `--transport http` | Modern HTTP clients, multiple concurrent connections |
| WebSocket | `--transport ws` | Real-time bidirectional, low-latency clients |

For Claude Desktop (local), stdio is the correct choice. For a remote server, use `sse` or `http`.

```sh
# Remote HTTP server
protomcp run tools.py --transport sse --host 0.0.0.0 --port 8080
```

## Call timeout

The default call timeout is 5 minutes. Adjust for your workload:

```sh
# Long-running tasks
protomcp run tools.py --call-timeout 30m

# Fast tools only
protomcp run tools.py --call-timeout 30s
```

## Logging

```sh
protomcp run tools.py --log-level debug   # debug, info, warn, error
```

Logs are written to stderr in structured format.

## systemd

Create `/etc/systemd/system/protomcp.service`:

```ini
[Unit]
Description=protomcp MCP server
After=network.target

[Service]
Type=simple
ExecStart=/usr/local/bin/protomcp run /opt/mytools/tools.py --transport sse --host 127.0.0.1 --port 8080
WorkingDirectory=/opt/mytools
User=protomcp
Restart=on-failure
RestartSec=5
StandardOutput=journal
StandardError=journal

[Install]
WantedBy=multi-user.target
```

Enable and start:

```sh
sudo systemctl daemon-reload
sudo systemctl enable protomcp
sudo systemctl start protomcp
```

## Docker

```dockerfile
FROM python:3.12-slim

# Install protomcp binary
RUN curl -L https://github.com/protomcp/protomcp/releases/latest/download/protomcp_linux_amd64.tar.gz \
    | tar xz -C /usr/local/bin/

# Install Python SDK and tool dependencies
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY tools.py .

EXPOSE 8080
CMD ["protomcp", "run", "tools.py", "--transport", "sse", "--host", "0.0.0.0", "--port", "8080"]
```

`requirements.txt`:

```
protomcp
```

Build and run:

```sh
docker build -t mytools .
docker run -p 8080:8080 mytools
```

## Unix socket path

By default, protomcp creates the unix socket at `$XDG_RUNTIME_DIR/protomcp/<pid>.sock`. Override with `--socket`:

```sh
protomcp run tools.py --socket /tmp/mytools.sock
```

This is useful when you have multiple tool processes and want predictable socket paths.

---

## Multiple tool processes

Run separate protomcp instances for each tool file:

```sh
protomcp run auth_tools.py --transport sse --port 8081 &
protomcp run data_tools.py --transport sse --port 8082 &
```

Configure each as a separate MCP server in your client.

## Health monitoring

protomcp does not expose a health endpoint, but you can monitor the process directly:

```sh
# systemd
systemctl status protomcp

# Docker
docker inspect --format='{{.State.Health.Status}}' mytools-container
```

For HTTP transports, a successful `tools/list` call indicates the server is healthy.
